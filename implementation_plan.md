### Idea 1: Hybrid Graph-Scoring (TextRank + Domain Contrast)
**The Concept:** Standard keyword extraction often uses an unsupervised graph algorithm called **TextRank** (which is based on Google's PageRank). It builds a network of words and draws edges between words that co-occur in the same sentence. 
**The Novelty:** Standard TextRank treats all words equally. What if we fuse TextRank with `pslegal`? We can modify the TextRank algorithm so that the network *edge weights* are mathematically multiplied by the `PSLegal` domain-contrast score. 
**Why it’s great:** You mathematically marry two vastly different AI philosophies: topological graph theory (TextRank) and statistical domain-contrast (`pslegal`). This is highly academically rigorous and has not been done in the provided papers.

### Idea 2: Unsupervised Semantic Extraction (Legal-KeyBERT)
**The Concept:** The paper relies entirely on "discrete mathematical counting" (exact string matching and frequency counting). Modern NLP is built on dense semantic embeddings (transformers). 
**The Novelty:** We can implement a zero-shot, unsupervised framework inspired by `KeyBERT`. We pass the document through a pre-trained `LegalBERT` or `SentenceTransformer` to get a "Document Vector." We then pass all Noun Phrases extracted by `extract_noun_phrases.py` through the same transformer to get "Phrase Vectors." We extract the phrases that have the highest cosine similarity to the total Document Vector.
**Why it’s great:** You can cleanly benchmark the outdated 2017 statistical methodologies (`pslegal`) against modern 2024 dense-vector Transformer approaches on the exact same dataset.

### Idea 3: Bridging Track 1 and Track 2 (Catchphrase-driven Precedent Retrieval)
**The Concept:** The `FIRE2017` dataset you provided actually has two folders: `Task_1` (Catchphrase Extraction) and `Task_2` (Precedence Retrieval - given a current case, find similar past cases).
**The Novelty:** Right now, catchphrases are extracted in a vacuum. Let's prove their *utility*. We can build an Information Retrieval pipeline where we use the catchphrases extracted by `pslegal` as the **search queries** to find matching prior cases from `Task_2`. 
**Why it’s great:** We can measure if querying a database using *just massive legal documents* performs worse or better than querying the database using *only the 10 Catchphrases* extracted by your algorithm. This bridges both modules of the dataset into a single cohesive pipeline!

### Idea 4: LLM-as-a-Judge for Unsupervised Filtering
**The Concept:** The main flaw of unsupervised logic is that it generates garbage nouns that happen to beat the statistical math. 
**The Novelty:** We use `pslegal` as a high-recall filter to extract 50 potential catchphrases. Then, we use a lightweight open-source LLM API (like Llama-3 or Mistral via a free API endpoint) and prompt it to act as a "Legal Expert." The LLM's only job is to binary classify (Keep/Discard) the 50 phrases generated by `pslegal` into a polished final list.
**Why it’s great:** LLMs are great but computationally too slow/expensive to read 100 massive legal cases. `pslegal` is extremely fast but noisy. By piping `pslegal` output into an LLM, you combine the speed of unsupervised statistical parsing with the cognitive reasoning of generative AI.
